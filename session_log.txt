2. Forecasting: Hybrid — rule-based baseline + feature-weighted gradient boost (Recommended)

Why:
- Pure rule-based feels toy-ish; judges will dock the 20% Technical Excellence
- Full LSTM is overkill for a hackathon and risks demo failure if training misbehaves
- A gradient-boosted model on synthetic-but-realistic time series gives you:
  - Transparent feature importance (defensible to judges: "wind direction contributed 34%")
  - 24-72h predictions with confidence intervals
  - Comparison vs naive baseline (persistence = "tomorrow = today's AQI") → instantly shows value
- FastAPI can serve predictions in <50ms → smooth demo

Architecture:
features = [current_aqi, hour, day_of_season, traffic_index,
            wind_speed, wind_dir_to_industrial, temp, humidity,
            stubble_burning_proxy, construction_intensity]

        GradientBoostingRegressor → predicted_aqi_24h, 48h, 72h

3. Source Attribution: Weighted Geospatial Heuristic Engine with Confidence Scoring (Recommended)

Why:
- Fully explainable → every score has a clear "why" → judges love this
- No training data dependency (we're seeding anyway)
- Maps directly to the 4 categories from your ReadMe: traffic, construction, industrial, waste-burning
- Outputs ranked sources + confidence + supporting signals exactly as specified

Approach:
For each ward polygon:
1. Compute proximity-weighted scores for each source type:
  - Traffic: road density within ward + distance to major highways
  - Construction: construction site count + active project intensity
  - Industrial: distance to industrial zones (weighted by emission type)
  - Waste burning: reported incidents + seasonal multiplier (post-harvest, winter)
  - Stubble burning (Delhi-specific): wind-direction exposure to Punjab/Haryana
2. Apply wind-direction modifiers (downwind sources contribute more)
3. Normalize → confidence-weighted ranking
4. Return top-2 sources + confidence + human-readable explanation

1. Solution Strategy

Problem Framing

India's air pollution crisis is documented (CPCB data, 900+ CAAQMS stations, 1.67M annual deaths per Lancet) but the intelligence layer to act on it doesn't exist. City admins see AQI numbers; they don't know what's causing it, what it'll be tomorrow, or what to do about it.

Our Differentiation

We're not an AQI dashboard. We're an intervention decision-support system.

The 4-question loop every city admin needs answered, in 5 seconds:
1. Where is pollution worst right now? → Hotspot map
2. Why is it bad here? → Multi-modal source attribution with confidence
3. What will it be in 24/48/72h? → Atmospheric-dispersion-inspired forecast
4. What should I do about it? → Prioritized enforcement + health advisory

Plus the cross-city layer: "How is this compared to peer cities, and what worked elsewhere?"

Innovation Pillars (for the 25% Innovation score)

- Multi-modal attribution agent combining spatial, satellite, traffic, and land-use signals with per-agent evidence
- Atmospheric-dispersion-inspired features (not full CFD, but physics-shaped feature engineering)
- Intervention feedback loop — track past actions and their measured AQI impact
- Multi-city comparative learning — propagate what works across cities
- Vulnerability-weighted health advisories (not just AQI → text, but AQI + nearby sensitive populations)

5. API Contract (Summary — full version in docs/api-contract.md)

Cities & Overview

- GET /api/cities — list cities with current AQI summary
- GET /api/cities/{code}/overview — full city dashboard payload
- GET /api/cities/{code}/hotspots — ranked hotspot list
- GET /api/wards/{ward_id} — ward detail (vulnerability, sources nearby)

Map & Geo

- GET /api/cities/{code}/geo/wards — ward polygons (GeoJSON FeatureCollection)
- GET /api/cities/{code}/geo/layers/{layer} — institutions/industry/construction/thermal (GeoJSON)
- GET /api/cities/{code}/geo/traffic — traffic density grid

AI Engines

- GET /api/wards/{ward_id}/attribution — top sources + confidence + per-agent evidence
- GET /api/wards/{ward_id}/forecast?horizons=24,48,72 — forecast with confidence bands
- GET /api/wards/{ward_id}/forecast/compare — forecast vs persistence baseline

Action

- GET /api/cities/{code}/enforcement/queue — prioritized inspector action list
- GET /api/cities/{code}/enforcement/{ward_id} — specific intervention recommendation
- GET /api/cities/{code}/health/overlay — vulnerability overlay
- GET /api/wards/{ward_id}/advisory?lang=hi|en|kn|ta — multilingual advisory

Comparative

- GET /api/compare/cities?metric=aqi|intervention|trend — cross-city comparison
- GET /api/compare/interventions — what worked where

System

- GET /api/health — health check
- GET /api/demo/scenario — pre-canned demo flow data
- GET /api/demo/story — guided story mode steps

Response Shape (consistent envelope)

{
  "data": { ... },
  "meta": { "city": "DEL", "generated_at": "...", "model_version": "..." },
  "warnings": []
}

8. Key Assumptions

1. Demo data, not live feeds — internal consistency > real-time accuracy for judging
2. SQLite for demo — Postgres via DATABASE_URL env, schema is Postgres-compatible
3. No external API keys needed — fully self-contained, can be demoed offline
4. Ward polygons are simplified — real boundaries are complex; we use representative shapes
5. Forecast "model" is gradient-boosted on seeded data — but the features are physics-shaped so the engine feels real
6. Attribution "agents" are Python functions — multi-modal in inputs, not in actual separate processes
7. Multilingual advisories are templated — interface designed for LLM swap-in but uses deterministic templates for demo reliability
8. Time series covers 30 days history + 72h forecast — enough to show trends, short enough to seed quickly
9. Vulnerability index is computed from institution density + population density, not pulled from census
10. Past interventions are seeded with realistic measured AQI deltas to enable the "what worked" comparative view

    """Return (coordinates, bbox) for a simple n-gon polygon around (lat, lon)."""
